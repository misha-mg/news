**Version:** 0.1

## 1. Overview

A personal news aggregator that surfaces daily IT/AI updates in one place. Once a day, a Claude Cowork agent fetches configured RSS feeds, generates summaries via the Claude API, and writes structured records into a Supabase database. The React frontend reads from Supabase in real time and presents a clean, filterable news feed. No manual effort required after initial setup.

---

## 2. Goals

- Receive a daily curated briefing on IT/AI without manual monitoring
- Browse news structured by category and source with filter controls
- Automate the full pipeline: collect в†’ process в†’ store в†’ display
- Keep infrastructure maintenance near zero

## 3. User

## 5. Data Layer вЂ” Supabase

### 5.1 Why Supabase

- Managed PostgreSQL with a generous free tier
- Built-in REST and realtime APIs вЂ” no backend code needed
- Row-level security for future auth
- Clean migration path if volume grows

### 5.2 Database Schema

**Table: `articles`**

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` | Primary key, default `gen_random_uuid()` |
| `title` | `text` | Article headline |
| `url` | `text` | Original article URL, unique |
| `source` | `text` | Human-readable source name, e.g. `"Anthropic Blog"` |
| `category` | `text` | One of: `ai-release`, `engineering`, `tools`, `industry` |
| `published_at` | `timestamptz` | Publication date parsed from RSS feed |
| `collected_at` | `timestamptz` | When agent inserted this row, default `now()` |
| `summary` | `text` | 2вЂ“3 sentence summary generated by Claude |

**Table: `digests`**

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` | Primary key |
| `date` | `date` | Digest date, unique constraint |
| `top_items` | `text[]` | Array of 5 headline strings |
| `summary` | `text` | "Key takeaways" paragraph written by agent |
| `created_at` | `timestamptz` | Default `now()` |

### 5.3 Indexes

```sql
-- Fast date-range queries for the feed
CREATE INDEX idx_articles_published_at ON articles (published_at DESC);

-- Filter by category
CREATE INDEX idx_articles_category ON articles (category);

-- Filter by source
CREATE INDEX idx_articles_source ON articles (source);
```

### 5.4 Row-Level Security

RLS enabled on both tables. In v1 a single anonymous read policy is sufficient:

```sql
-- Allow anonymous reads (frontend uses anon key)
CREATE POLICY "anon read articles" ON articles FOR SELECT USING (true);
CREATE POLICY "anon read digests" ON digests FOR SELECT USING (true);

-- Agent writes using service_role key вЂ” bypasses RLS by design
```

## 7. Agent (Claude Cowork)

**Schedule:** daily at 07:00  
**Credentials:** Supabase `service_role` key stored as a Cowork secret вЂ” never exposed to the frontend

**Step-by-step:**

1. Load RSS source list from config
2. For each source: HTTP GET в†’ parse XML в†’ extract `title`, `link`, `pubDate`, `description`
3. Discard items older than 24 hours
4. For each item: call Claude API to produce a 2вЂ“3 sentence English summary and assign one category from the allowed set
5. Deduplicate: `SELECT 1 FROM articles WHERE url = $1` вЂ” skip if already exists
6. Batch insert new articles into `articles` via Supabase REST
7. Generate daily digest: ask Claude to select top 5 articles and write a "key takeaways" paragraph
8. Upsert into `digests` (`ON CONFLICT (date) DO UPDATE`)
9. Log run: sources processed, articles inserted, failed sources

**Error handling:**

| Failure | Behavior |
|---|---|
| RSS feed unreachable | Log, skip source, continue |
| Claude API failure | Retry once; on second failure store article with `summary = null` |
| Supabase write failure | Log error, send Cowork notification |

### 8.5 Key Queries

  .from('articles')
  .select('*')
  .gte('published_at', new Date(Date.now() - 86400000).toISOString())

## 9. Deployment

### 9.1 Vercel

- GitHub repository connected to Vercel project
- Every push to `main` triggers a production deploy
- Build command: `vite build`
- Output directory: `dist`

### 9.2 No git push from agent

The agent writes directly to Supabase. The frontend fetches live data on every page load. Vercel redeploys only when frontend code changes вЂ” not when new articles arrive. This keeps the repo clean and the deploy pipeline simple.

## 11. Decisions

| Question | Decision |
|---|---|
| Summary language | **English** вЂ” all summaries and digest text generated in English |
| Article cap per day | **50 articles max** вЂ” agent stops inserting once the daily count hits 50, prioritizing sources by table order |
| Public access in v1 | **Yes** вЂ” site is publicly accessible from day one, no auth gate |
| Agent schedule | **Daily at 06:00 CET (UTC+1 / UTC+2 DST)** вЂ” Cowork cron configured as `0 6 * * *` Europe/Paris |
